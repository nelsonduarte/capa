# Empirical micro-validation: the capability-aware SBOM in practice

A small, fully reproducible comparison of a common
microservice pattern (config loading with disk + environment +
remote-flag overrides) in two forms:

- [`examples/empirical_config_naive.py`](../examples/empirical_config_naive.py),
  a typical Python implementation: one `load_config(path) ->
  dict` function that conflates filesystem, environment-variable,
  and HTTP access.
- [`examples/empirical_config.capa`](../examples/empirical_config.capa),
  the Capa equivalent: the same logic, split into functions
  whose signatures declare exactly the capabilities they need.

The pair is the smallest possible artefact for the empirical
claim in the thesis: **the capability-aware SBOM is a strict
information gain over a PURL-only SBOM**. The naive Python
version is real-world-shaped (any number of microservices have
exactly this loader pattern); the Capa version is what the
same logic looks like under the discipline; and the SBOMs the
two generate are directly comparable.

---

## The Python version

The full source is in
[`examples/empirical_config_naive.py`](../examples/empirical_config_naive.py).
The signature is:

```python
def load_config(path: str) -> dict:
    ...
```

The body, in three steps, reads from disk, overlays
environment-variable overrides, and (conditionally) fetches a
JSON patch from a remote URL. Three different capabilities,
one signature, no indication in the function header of which
authorities are involved.

A reviewer reading this code in a pull request gets the three
capabilities from the imports at the top of the file (`json`,
`os`, `urllib.request`) but no per-function attribution. An
auditor reading a CycloneDX SBOM generated by, say, `syft` or
`pip-licenses`, gets:

```jsonc
{
  "components": [
    { "type": "library", "name": "json", ... },
    { "type": "library", "name": "os",   ... },
    { "type": "library", "name": "urllib", ... }
  ]
}
```

A list of stdlib modules. No information about which functions
in *this* application use which of them.

If a malicious update to this file added a fourth step (POST
the merged config to an attacker's host), the change would not
be visible in any signature; it would not be visible in the
SBOM, since the SBOM was already saying "this code might touch
the network". The "might" is doing all the work.

## The Capa version

The full source is in
[`examples/empirical_config.capa`](../examples/empirical_config.capa).
The same logic is split into five functions:

| Function | Signature shape | Declared capabilities |
|---|---|---|
| `parse_config_text` | `(String) -> Config` | *(none)* |
| `has_field` / `set_field` | pure helpers | *(none)* |
| `load_local_config` | `(Fs, String) -> Result<Config, IoError>` | `Fs` |
| `apply_env_overrides` | `(Env, Config) -> Config` | `Env` |
| `fetch_remote_overrides` | `(Net, String) -> Result<Config, IoError>` | `Net` |
| `load_full_config` | `(Fs, Env, Net, String, String) -> Result<Config, IoError>` | `Fs`, `Env`, `Net` |
| `main` | `(Stdio)` | `Stdio` |

Each capability appears in the parameter list of every
function that touches it. There is no way for
`load_local_config` to also read the environment: the body
cannot reference `env` because no `Env` parameter is in scope.
The type checker rejects any attempt.

## The SBOM diff

Generated by `capa --cyclonedx examples/empirical_config.capa`,
extracted with one line of Python:

```
parse_config_text:      []
has_field:              []
set_field:              []
load_local_config:      ['Fs']
apply_env_overrides:    ['Env']
fetch_remote_overrides: ['Net']
load_full_config:       ['Fs', 'Env', 'Net']
main:                   ['Stdio']
```

Eight functions, eight per-function entries. A reviewer with
this SBOM in hand knows:

- **What is pure.** `parse_config_text`, `has_field`,
  `set_field`. Zero capability declarations. The compiler
  has verified that the body of each cannot touch any
  ambient authority.

- **What needs each single capability.** `load_local_config`
  needs `Fs` and only `Fs`. If a future revision quietly adds
  a network call to this function, the SBOM entry changes
  from `['Fs']` to `['Fs', 'Net']`. The widening is loud.

- **What the full authority surface is.** `load_full_config`
  carries `Fs + Env + Net`. The caller of this function
  knows, from the signature alone, that invoking it grants
  the function three distinct authority dimensions. The
  caller can choose to pass attenuated handles
  ([`fs_env_attenuation.capa`](../examples/fs_env_attenuation.capa))
  to bound each.

A PURL-based SBOM for the Python version cannot say any of
this. It can list the *imports* of the application; it cannot
attribute per-function authority. The capability-aware SBOM is
a strict information gain.

## Why this is the empirical claim

The argument in [`docs/cra.md`](cra.md) is that Annex I Part
II (1) of the CRA (SBOM in a machine-readable format) is
satisfied by *any* CycloneDX or SPDX output, but the
capability-aware variant Capa emits is **strictly more
informative** for the integrity arguments Annex I Part I
(secure-by-default, attack-surface minimisation, data
minimisation, exploitation mitigation) requires.

This file is the smallest demonstration of that claim.
Everything is reproducible from the repo:

```bash
# The Python version (for reading; not a runnable demo):
cat examples/empirical_config_naive.py

# The Capa version (runs the in-memory demo):
python -m capa --run examples/empirical_config.capa

# The capability-aware SBOM:
python -m capa --cyclonedx examples/empirical_config.capa
```

The Capa side runs in 50 ms; the SBOM emits in another 50 ms.
A thesis chapter that cites this comparison can do so with one
artefact, one command, one reproducible diff.

## What this does *not* claim

Three honest scoping notes:

1. **One example is not a study.** This file shows the
   mechanism. A real empirical study would transliterate
   multiple real-world libraries (preferably ones with known
   capability creep across versions) and report quantitative
   results. That is on the thesis backlog; this is the
   micro-validation.

2. **The Capa version is hand-written.** The transliteration
   is not automatic. Future work on a Python-to-Capa
   transpilation tool would let the comparison happen at
   scale; for now, the per-function capability sets are the
   compiler's verification of human-written code.

3. **The naive Python version is illustrative.** Many Python
   developers do write more disciplined code (passing
   handles explicitly, using dependency injection). The
   comparison here is against the *typical* shape, not
   against the best-engineered shape. Even disciplined
   Python lacks the static guarantee Capa provides: a
   well-engineered Python function that *says* it only
   touches Fs still cannot be checked at compile time.
